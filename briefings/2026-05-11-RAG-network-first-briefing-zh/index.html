今日主题：RAG

今天的主题是 [[RAG]]。它仍然是当前 vault map 的 anchor topic，因为后面的 Agent、Evaluation、Acceleration、Online Learning 和 Mechanistic Interpretability 最后都会回到一个共同问题：外部证据如何被选择、压缩、消费、校验和保护。

今天的前沿变化不在“又多了一个更强 retriever”，而在 RAG 开始更明确地分成三条线：路由、多步控制、结构化证据充分性。换句话说，RAG 现在更像一个 query-aware、path-aware、budget-aware 的控制系统，而不是固定的“检索几段文本然后拼给模型”的流水线。

本周位置

Anchor topic。

周一固定回到 RAG，不是为了重复基础知识，而是为了用一个真实系统问题重新组织后续几天的阅读：哪些工作是在改进证据选择，哪些是在改进证据消费，哪些只是把测量噪声当成了进步。

本主题你现在需要知道的 3 件事

1. RAG 的核心问题正在从 retrieve more 转向 route better。 今天更关键的问题不是“召回更多文档”，而是“对这个 query、这个 corpus、这个成本预算，应该走哪一种 RAG 范式、哪一种 retriever、哪一种控制路径”。

2. 相关性不再等于生成效用。 现在更值得追的不是单一 Recall@k，而是 retrieved evidence 是否真的能支撑 generator 产出正确答案，尤其是在多轮、表格、法律、agentic 推理场景里。

3. GraphRAG 和 agentic RAG 正在合流。 GraphRAG 提供结构化路径与子图选择，agentic RAG 提供学习式控制和中间路径奖励；两者共同把 RAG 推向“可训练的证据推理控制”。

前沿变化观察

1. Routing 已经从概念变成 benchmark 和方法双线并进。

RAGRouter-Bench 先把 adaptive RAG routing 做成 benchmark；R^3AG 再把 retriever routing 写成可学习问题。这个顺序很重要，说明 routing 不再只是工程直觉，而是可比较的研究对象。

2. RAG 评估对象正在从最终答案转向中间路径。

Search-P1 和 AgenticRAGTracer 分别站在训练与诊断两端：前者给中间检索路径奖励，后者用 hop-aware benchmark 暴露多跳检索在哪一步坏掉。今天的强信号是“中间路径本身值得被训练和评估”。

3. 多轮对话 RAG 逐渐变成独立难题。

MTRAG、MTRAG-UN、LexRAG 这一簇说明，RAG 一旦进入对话，问题就不再只是检索相关文档，而是检索如何跟随用户意图变化、历史对话状态和不可回答轮次。

4. GraphRAG 的关键问题从“有没有图”转向“子图是否最小且充分”。

Retrieving Minimal and Sufficient Reasoning Subgraphs... 和 EHRAG 都说明，图结构本身不是答案；真正重要的是 query-conditioned subgraph sufficiency，以及结构与语义如何共同决定检索路径。

5. 社区层的真实痛点仍然是 silent failure。

last30days 的 Reddit-only 结果虽然弱，但 r/Rag / r/LangChain 最近反复出现两个表述：RAG 不会“崩”，而是会静默返回错答案；问题往往不在模型本身，而在 corpus、debug 和 repair layer。这和今天的论文线是同向的。

Read Now

1. RAGRouter-Bench: A Dataset and Benchmark for Adaptive RAG Routing

  • why now： 这是今天最适合作为主入口的一篇。它把 RAG routing 从经验工程问题变成了标准化 benchmark。
  • method / benchmark / mechanism point： 7,727 queries、21,460 documents，覆盖多域语料与三类 canonical query；统一比较五种 RAG paradigm 的生成质量与资源消耗，并强调 query-corpus compatibility。
  • how to read it first： 先读 query-corpus compatibility 的定义，再看五种范式是如何标准化比较的。重点不是谁第一，而是为什么没有一种范式在所有 query/corpus 组合上通吃。
  • source link： https://arxiv.org/abs/2602.00296

2. R$^3$AG: Retriever Routing for Retrieval-Augmented Generation

  • why now： 它正好补上 RAGRouter-Bench 的方法侧空白：如果需要 routing，路由器到底该学习什么信号。
  • method / benchmark / mechanism point： 把 retriever capability 拆成 retrieval quality 和 generation utility 两个可学习维度，并用文档评估信号与下游答案正确性一起做对比学习。
  • how to read it first： 先看作者如何证明“相关”不等于“对生成有用”，再看 query-specific preference shift 是怎么被监督到的。阅读目标是建立“语义相关性不是最终目标”的判断。
  • source link： https://arxiv.org/abs/2604.22849

3. Search-P1: Path-Centric Reward Shaping for Stable and Efficient Agentic RAG Training

  • why now： 如果今天只读一篇 agentic RAG 训练论文，就读这篇。它最明确地把学习信号从最终答案扩展到中间检索路径。
  • method / benchmark / mechanism point： 用 path-centric reward shaping 缓解稀疏奖励与失败样本无信号的问题，让 agentic RAG 在训练时直接得到中间轨迹质量反馈。
  • how to read it first： 先看 reward 设计，不要先看最终 accuracy。重点判断这些路径信号是否真的约束了检索推理，而不是在特定 benchmark 上偶然有效。
  • source link： https://arxiv.org/abs/2602.22576

4. AgenticRAGTracer: A Hop-Aware Benchmark for Diagnosing Multi-Step Retrieval Reasoning in Agentic RAG

  • why now： 它是 Search-P1 的天然配对阅读：一个教系统怎么学路径，一个教你怎么诊断路径在哪一跳坏掉。
  • method / benchmark / mechanism point： 构建 hop-aware benchmark，把最终多跳查询拆成可逐步验证的中间跳级问题，用来定位过早停止检索或盲目扩展搜索的问题。
  • how to read it first： 先看 hop-aware 构造方式，再看失败模式分类。阅读目标是形成“多步 RAG 不应只看 final answer”的诊断框架。
  • source link： https://arxiv.org/abs/2602.19127

5. Retrieving Minimal and Sufficient Reasoning Subgraphs with Graph Foundation Models for Path-aware GraphRAG

  • why now： 这是今天最值得补的 GraphRAG 论文，因为它把图检索问题明确收紧到“最小且充分的子图”。
  • method / benchmark / mechanism point： 用预训练 Graph Foundation Model 直接做跨域 subgraph retrieval，再用 label-free subgraph selector 和 Information Bottleneck 目标找 query-conditioned 的 minimal, sufficient core set。
  • how to read it first： 先读 GFM-Retriever 如何直接返回 subgraph，再看 Information Bottleneck 为什么能约束“既不缺证据，也不结构冗余”。这比泛泛讲 GraphRAG 更贴近当前主线。
  • source link： https://arxiv.org/abs/2603.07179

Watchlist

  • MTRAG: A Multi-Turn Conversational Benchmark for Evaluating Retrieval-Augmented Generation Systems

多轮 RAG 的基础基准。今天不放 Read Now，是因为本轮更强信号在 routing 和 path-level control；但如果下周要继续跟对话式 RAG，这是优先级很高的背景入口。

  • EHRAG: Bridging Semantic Gaps in Lightweight GraphRAG via Hybrid Hypergraph Construction and Retrieval

适合放在 GraphRAG watchlist。它的价值是把轻量级图结构从纯共现推进到结构+语义混合超图，尤其适合低索引成本场景。

  • From BM25 to Corrective RAG: Benchmarking Retrieval Strategies for Text-and-Table Documents

本地主线必读，但今天先放 watch。它最重要的结论是 dense retrieval 并不天然优于 BM25，尤其在图文表混合和精确数值问题上。

  • RAGChecker: A Fine-grained Framework for Diagnosing Retrieval-Augmented Generation

仍是细粒度诊断入口。与 AgenticRAGTracer 的区别是它偏模块级与系统级 error breakdown，而不是 hop-aware 多步路径。

  • RAG Security

今天没有发现足以压过 routing 主线的新安全论文信号，但 HubScan、CPA-RAG、Disabling Self-Correction 仍应保留。当前更合理的做法是把安全线当作稳定 watchlist，而不是硬造“今天有新变化”。
