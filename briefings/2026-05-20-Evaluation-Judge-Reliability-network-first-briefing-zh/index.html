今日主题：Evaluation / Judge Reliability

今天是周三，轮转到 [[Evaluation]] / [[Judge Reliability]]。05-13 那份已经覆盖了 Bias and Uncertainty、False Trust 和 JudgeSense：核心结论是 judge reliability 不能只看模型打分，calibration estimator、解释呈现方式和 prompt protocol 都会制造误差。

今天的新增量更进一步：judge reliability 正在从“协议层诊断”下沉到“机制层”和“约束层”。也就是说，问题不只是 judge 输出受 prompt 影响，而是模型内部可能存在一个抽象评判信号，再通过脆弱的输出格式分支被投射成分数、True/False 或 pairwise verdict；同时，整体回答看起来合格，并不代表每个具体约束都被可靠检查。

本周位置

Adjacent topic。

Evaluation / Judge Reliability 是 RAG、Agent 和 Deep Research 的共同测量层。RAG 需要判断答案是否被证据支持，Agent 需要判断任务是否真的完成，Deep Research 需要判断引用和组织是否可靠；这些判断最后都可能落到 judge、rubric、verifier 或 human review protocol 上。

和 RAG 的关系

RAG 的质量经常被一个 evaluator 压缩成“faithful / grounded / helpful”之类的标签。但今天的信号提醒我们：如果 evaluator 的输出格式、约束拆分方式或内部评判电路本身不稳定，那么 RAG 的评估结论可能是在测 judge protocol，而不是测检索和证据质量。

本主题你现在需要知道的 3 件事

1. 输出格式可能在污染评判结果。

同一个模型用 1-5 分、True/False 或偏好选择来做 judge，可能不是同一件事。Judge Circuits 暗示 benchmark 跨格式比较时，可能部分测到了 formatter geometry（输出格式几何），而不是纯粹的 evaluation quality。

2. 整体判断不足以诊断复杂任务。

MCJudgeBench 把 judge evaluation 拆到 constraint level（逐约束层面）：每个响应需要满足多个要求，每个要求都有 yes / partial / no 标签。结果显示，高总体正确率不保证低不一致性，reasoning 也不稳定地提升 judge stability。

3. 多轮协作评估需要轨迹级信号。

对 coding / co-creation 这类人机协作任务，只看最终产物太粗。新的评估框架开始同时看 schema-constrained judge output、probabilistic reliability、inter-judge agreement 和 turn-wise trajectory signals。

What I Learned

  • 05-15 的 Judge Circuits 是今天最强的新论文。它把 judge 不一致性从输入输出现象推进到内部机制：开放权重模型里可能有一个稀疏的 Latent Evaluator 子图负责抽象评判，而输出格式分支负责把这个连续信号映射成具体标签。
  • MCJudgeBench 补上了一个更工程化的缺口：judge 要能逐条检查指令约束，而不是只给整体回答打分。它的关键结论是 correctness 和 inconsistency 是两条轴。
  • coding co-creation 的 reliability-aware framework 说明实际评估协议会越来越像实验系统：要防 trajectory leakage，要修复 schema 输出，要报告 Brier score / ECE / kappa 这类可靠性指标，还要看多轮交互轨迹。

Key Patterns

1. 从 protocol sensitivity 到 internal mechanism。

之前我们知道 prompt wording、rubric 和 output format 会改变 verdict；Judge Circuits 进一步问：模型内部到底哪里在做“评判”，哪里在做“格式化输出”。

2. 从 whole-response judge 到 constraint-level judge。

多约束任务的失败往往不是“整题错”，而是某些 constraint partial / no。只看整体标签会掩盖稀有类别和局部约束的可靠性问题。

3. 从单次打分到 reliability metrics bundle。

新框架开始把 ROC-AUC、PR-AUC、MCC、LogLoss、Brier score、ECE、Cohen's kappa / Fleiss' kappa 放在一起，而不是只报告 judge agreement 或平均分。

4. 从解释性输出到可审计评估链。

05-13 的 False Trust 已经说明解释会提高说服力但不一定提高判断力；今天的增量说明，真正可用的评估链要把 schema、validation、repair、leakage control 和 per-instance reliability 放进协议。

Read Now

1. Judge Circuits

  • why now: 这是 05-13 之后最强的新信号。它不再只是说 judge 在不同格式下会不一致，而是用 Position-aware Edge Attribution Patching（位置感知边归因修补，PEAP）追踪内部机制。
  • what changed: 论文在 Gemma-3、Qwen2.5 和 Llama-3 中发现一个稀疏的 Latent Evaluator 子图；清零这个子图会让评判能力崩塌，但在部分结构模块化模型里保留世界知识。它把“抽象评判”和“输出格式化”拆开看。
  • read it for: 如果你想把 Judge Reliability 接到 Mechanistic Interpretability，这篇是桥。它说明 benchmark 里的跨格式可靠性比较，可能部分是在测输出格式分支。
  • how to start: 先看 format-induced inconsistency 的例子，再看 Latent Evaluator subgraph 和 terminal branches 的定义，最后看 zero-ablation 结果。

2. MCJudgeBench: A Benchmark for Constraint-Level Judge Evaluation in Multi-Constraint Instruction Following

  • why now: 它把 judge 评估从整体回答拆到单个约束，这对 RAG 和 Agent 都更真实：复杂任务通常是多个条件同时成立，而不是一个总标签能概括。
  • what changed: 每个样本包含 instruction、candidate response、explicit constraint list、per-constraint gold labels，并加入 response perturbations 和 prompt variants。它区分 intrinsic inconsistency 和 procedural inconsistency。
  • read it for: 用来设计本地 evaluator harness 的最小稳定性检查：既要看 correctness，也要看不同标签类别、prompt variants 和 perturbations 下的一致性。
  • how to start: 先读 yes / partial / no 标签体系，再看“reasoning improves correctness but not uniformly stability”的结果。

3. LLM-as-a-Judge for Human-AI Co-Creation: A Reliability-Aware Evaluation Framework for Coding

  • why now: 这篇不是纯 judge theory，但它给了一个很实用的多轮 coding / co-creation 评估模板。对 Codex 式任务尤其相关。
  • what changed: 它把 schema-constrained judge outputs、validation and repair、user/problem split、防 trajectory leakage、probabilistic reliability 和 inter-judge agreement 放进同一协议，还追踪 Success-at-Turn、time-to-success、revision churn 等轨迹信号。
  • read it for: 如果你要评估人机协作而不是单轮答案，这篇比单一 benchmark 更接近真实工作流。
  • how to start: 先看 judge protocol 的指标包，再看为什么 inter-judge consistency 仍然 modest。

Watchlist
