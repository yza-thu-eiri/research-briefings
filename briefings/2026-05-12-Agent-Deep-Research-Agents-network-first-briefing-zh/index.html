今日主题：Agent / Deep Research Agents

今天是周二，轮转到 [[Agent]] / [[Deep Research Agents]]。这条线对 vault map 重要，不是因为“agent”这个词还热，而是因为 deep research 已经开始逼着系统回答三个更硬的问题：怎样评测研究过程、怎样把 verification 前移到 agent 设计里、以及怎样让检索真正读懂 agent 的推理状态。

这也是它和一般 report generator 的分界线。现在的前沿，不再满足于“写出长报告”，而是开始追问报告背后的搜索路径、证据核验、引用链条和停止行为是不是可诊断、可复查、可优化。

本周位置

Adjacent topic。

本周锚点仍然是 RAG；Agent / Deep Research Agents 是把 RAG 放进真实研究工作流后的近邻主题。它最有价值的地方，不是替代 RAG，而是把 RAG 的三个薄弱处暴露得更彻底：检索是否足够全面、证据是否能核验、系统是否知道什么时候该停。

和 RAG 的关系

Deep research agent 可以看成“长程、带报告目标、带引用责任的 RAG”。普通 RAG 往往回答单个问题；deep research agent 要先拆任务、再发搜索、再合并来源、再判断覆盖是否够、最后还要把每条关键断言放回引用链里。

所以今天最重要的变化，不是 agent 会不会多调用几个工具，而是 RAG 里的 retrieval、verification 和 stopping 三个问题，正在被单独拉成研究对象。

本主题你现在需要知道的 3 件事

1. 评测中心正在从“最终报告像不像”转向“研究过程好不好”。

最新一波工作开始同时看过程审计（process audit）、事实核验（factuality verification）和多模态证据，而不是只给一份长报告打总分。

2. verification 正在从 evaluator 的事，前移成 agent 设计本身的一部分。

这意味着“先生成、后检查”的旧套路开始不够了；训练数据、轨迹构造和 test-time scaling 都在引入显式核验环。

3. deep research retrieval 已经开始脱离 query-only 范式。

新检索器不再只看搜索字符串，而是把 agent 搜索前写下来的 reasoning trace 一起当检索输入。这对 RAG 很关键，因为很多召回失败其实不是文档不存在，而是 query 表达丢掉了意图和阶段信息。

What I Learned

  • 2026 年 3 月这一波新增信号非常集中：MiroEval、DeepFact、DeepResearch Bench II 都在把 deep research 评测拆细，说明领域已经从“有没有 benchmark”进入“benchmark 怎么才不失真”的阶段。
  • 设计侧同步出现了 Marco DeepResearch 这样的 verification-centric agent，说明 verification 不再只是事后审查，而是 agent 主体能力的一部分。
  • 检索侧由 AgentIR 接上：deep research agent 的 reasoning state 本身正在变成检索信号，而不是噪声。
  • 社区层面的弱信号也和这个方向同向：大家开始更频繁地讨论 scaffold、验证步骤和 agent 透明度，而不只是比较最终跑分。

Key Patterns

1. 从 outcome-only 走向 process-aware evaluation。

MiroEval 强调过程和结果要一起看，尤其把多模态任务单独拉出来，说明“报告对了”已经不足以代表系统真的会研究。

2. 从 static benchmark 走向 evolving benchmark。

DeepFact 的核心不是又做了一个 fact benchmark，而是指出静态标注本身会失真，所以需要可修订、可审计的 benchmark 更新机制。

3. 从 post-hoc verification 走向 verification-centric design。

Marco DeepResearch 把 verification 放进数据合成、轨迹构造和 test-time scaling，等于把“核验”从尾部插件改成主干结构。

4. 从 query matching 走向 reasoning-aware retrieval。

AgentIR 表明 deep research agent 的搜索前推理不是冗余文字，而是最该进入 retriever 的高价值上下文。

5. 从总分评测走向可解释诊断。

DeepResearch Bench II 用 expert-report-derived rubrics 拆出 information recall、analysis、presentation，说明领域需要知道系统到底坏在“没找全”“没分析到位”还是“没写清楚”。

Read Now

1. MiroEval: Benchmarking Multimodal Deep Research Agents in Process and Outcome

  • why now： 它最像今天这波变化的总纲。不是再做一个总分 benchmark，而是把 process、factuality 和 multimodal outcome 一起拉进同一评测框架。
  • what changed： 相比更早的 deep research benchmark，它明显更强调“研究过程本身”是否可靠，而不只是最后报告像不像专家稿。
  • read it for： 如果你想判断 deep research agent 下一阶段会怎么被评，这篇最值得先读。
  • how to start： 先读三维 evaluation suite，再看 authors 总结出的三条 principal findings。

2. Marco DeepResearch: Unlocking Efficient Deep Research Agents via Verification-Centric Design

  • why now： 它代表设计线的新阶段：verification 不再只是 evaluator 的工作，而是 agent 训练与推理时的中心结构。
  • what changed： 核验机制被嵌进 QA data synthesis、trajectory construction 和 test-time scaling，说明 deep research agent 开始系统性处理“错误如何一路传播”。
  • read it for： 如果你更关心怎么把 deep research 做成稳定系统，而不是只看 benchmark，这篇优先级很高。
  • how to start： 先看三层 verification-centric design，再看它在 BrowseComp / BrowseComp-ZH 上的预算约束结果。

3. AgentIR: Reasoning-Aware Retrival for Deep Research Agents

  • why now： 这是今天最直接连接 RAG 的一篇。它不是泛泛讲 retrieval，而是明确说 deep research agent 的 reasoning trace 应该成为 retriever 输入。
  • what changed： query-only retrieval 被正面挑战；retriever 开始消费任务阶段、证据缺口和搜索意图。
  • read it for： 如果你想把 deep research topic 重新映射回本地 RAG Retrieval Recall 主线，这篇是桥。
  • how to start： 先看 Reasoning-Aware Retrieval 和 DR-Synth 的定义，再看它相对 BM25 和 conventional embeddings 的增益。

Watchlist

  • DeepFact: Co-Evolving Benchmarks and Agents for Deep Research Factuality

很重要，但更适合紧跟在 MiroEval 后读。它真正新的是 Audit-then-Score 思路：benchmark 本身也要在争议中被修订，而不是假装标签一开始就稳固。

  • DeepResearch Bench II: Diagnosing Deep Research Agents via Rubrics from Expert Report

适合补 report-quality 诊断层。它把“报告差”拆成 recall、analysis、presentation 三维，可解释性比只看总分强很多。

  • Gemini Deep Research Agent

这是产品侧信号，不是论文。它的重要性在于 deep research agent 已经被做成 Interactions API 中的异步 agent primitive，说明产业界默认这类任务会是长时、后台、可轮询的工作流。
