2026-05-04 RAG 研究简报

今日主题

今日主题是 RAG（检索增强生成，Retrieval-Augmented Generation）。它是当前 vault map 的 anchor topic：一方面连接 GraphRAG、RAG Security、多轮对话 RAG 和 agentic RAG；另一方面也是判断长上下文、工具调用、智能体检索和评测体系是否真正有效的共同入口。

今天的前沿信号比较强。变化不在“又多了一个更强 embedding model”，而在 RAG 开始被重新定义为一个路由和控制问题：系统要决定何时检索、用哪种检索范式、检索结果是否真的能支撑生成、以及多步检索过程如何训练和诊断。

本周位置

本周位置：anchor topic。周一固定回到 RAG，是为了把后续 Agent、Evaluation、Acceleration、Online Learning 和 Findings 的读法锚回一个实际系统问题：外部证据如何被选择、压缩、消费、评测和保护。

本主题你现在需要知道的 3 件事

1. RAG 的核心问题正在从 retrieve more 转向 route better。也就是说，检索不是默认开启的单一流水线，而是要根据 query、corpus、成本和生成目标选择范式。

2. Recall@k 仍然重要，但已经不够。今天更有价值的指标是证据充分性、生成可用性、query-corpus compatibility（查询和语料的适配度）以及多步检索路径是否合理。

3. GraphRAG 和 agentic RAG 正在合流。GraphRAG 提供结构化证据路径，agentic RAG 提供动态决策能力；两者共同把 RAG 从“文本拼接”推向“可训练的检索推理控制”。

What Changed

  • 自适应路由成为主线。RAGRouter-Bench 把 RAG 范式选择定义成 query-corpus compatibility 问题，说明没有一种 RAG pipeline 在所有查询和语料组合上都最优。
  • 检索器选择不再只看相关性。R^3AG 把 retriever capability 拆成 retrieval quality 和 generation utility：文档不仅要相关，还要能真正帮助生成器答对。
  • 表格和半结构化文档继续暴露稠密检索的边界。From BM25 to Corrective RAG 显示，在金融图文表混合文档上，BM25 和 hybrid pipeline 仍然有很强现实价值。
  • agentic RAG 的训练目标开始从最终答案扩展到路径质量。Search-P1 用 path-centric reward shaping 给中间检索和推理轨迹提供更密集的学习信号。
  • 社区信号弱于论文信号。last30days 这轮只有 Reddit 40% 覆盖，且多数是噪声；可保留的真实信号主要是 r/Rag / r/LocalLLaMA 对本地文档检索、embedding 和缓存效率的零散讨论，不足以主导读文决策。

Read Now

1. RAGRouter-Bench: A Dataset and Benchmark for Adaptive RAG Routing

  • why now：这是今天最适合作为 RAG 主线入口的一篇。它把 RAG 从“选择更强方法”改写成“按查询和语料选择合适范式”的 benchmark 问题。
  • method / benchmark / mechanism point：构造 7,727 个查询和 21,460 个文档，覆盖多领域语料和三类查询，并同时评估生成质量与资源消耗。
  • how to read it first：先看 query-corpus compatibility 的定义，再看五种 RAG paradigm 怎么标准化，最后看哪些场景下高级机制反而不划算。
  • source link：https://arxiv.org/abs/2602.00296

2. R^3AG: Retriever Routing for Retrieval-Augmented Generation

  • why now：它正好补上 RAGRouter-Bench 的方法侧问题：如果知道 RAG 需要路由，路由器到底该学什么信号。
  • method / benchmark / mechanism point：把检索器能力拆成两个可学习维度：retrieval quality（检索质量）和 generation utility（生成效用），并用文档评估和下游答案正确性做对比学习监督。
  • how to read it first：先读问题设定，重点检查它如何证明“语义相关”不等于“对生成有用”；然后看不同 query 对 retriever 的偏好如何变化。
  • source link：https://arxiv.org/abs/2604.22849

3. From BM25 to Corrective RAG: Benchmarking Retrieval Strategies for Text-and-Table Documents

  • why now：这是本地 RAG 主线里最贴近“检索质量不等于回答质量”的实证论文，尤其适合用来校正对 dense retrieval 的过度信任。
  • method / benchmark / mechanism point：在 23,088 个金融 QA 查询和 7,318 份图文表混合文档上比较十类检索策略；hybrid retrieval + neural reranking 整体最强，但 BM25 在金融文档上能击败不少稠密检索。
  • how to read it first：先看 retrieval metrics 和 Number Match 的差距，再看 HyDE / multi-query / adaptive retrieval 在精确数值问题上的有限收益。
  • source link：https://arxiv.org/abs/2604.01733

4. Search-P1: Path-Centric Reward Shaping for Stable and Efficient Agentic RAG Training

  • why now：如果 RAG 正在走向 agentic control，这篇是理解“如何训练检索过程”而不是只评估最终答案的关键入口。
  • method / benchmark / mechanism point：提出 path-centric reward shaping，用路径覆盖、软评分和双轨路径评分给失败样本也提供学习信号；论文报告相对 Search-R1 平均准确率提升 7.7 points。
  • how to read it first：先看 reward 设计，不要先看榜单；重点判断中间路径信号是否真的约束了检索推理，而不是只在特定 QA benchmark 上调参成功。
  • source link：https://arxiv.org/abs/2602.22576

Watchlist

  • Lightweight Query Routing for Adaptive RAG：它用 RAGRouter-Bench 做轻量分类器 baseline，值得 watch 而不是立刻精读。技术价值在于给“便宜路由器是否够用”提供下限，但它主要是 query-side routing，尚未真正解决 corpus-aware routing。
  • AgenticRAGTracer：本地队列中优先级较高。它把多步 RAG 拆成 hop-aware 诊断，适合和 Search-P1 对读：一个问训练路径如何给 reward，一个问评测时如何定位哪一跳失败。
  • MTRAG：多轮对话 RAG 的基础 benchmark。今天不放 Read Now，是因为本轮更强信号在路由和控制；但如果后续要把 RAG 接到真实助手工作流，MTRAG 仍是必读背景。
  • RAG Security：本轮没有发现足以压过路由主线的新安全论文信号，但本地主题页里的 Bias Injection、CPA-RAG、HubScan 仍应保留。安全问题的关键不是生成器 guardrail，而是语料、向量空间和检索拓扑是否可被污染。
  • GraphRAG 自动演化：EvoRAG 这类 KG-RAG feedback-driven evolution 工作值得继续盯，但今天还不建议立刻精读；先把 RAG routing 和 GraphRAG-R1 / Search-P1 的控制主线读通。
