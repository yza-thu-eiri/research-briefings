今日主题：Evaluation / Judge Reliability

这是 2026-05-13 的补生成。当天是周三，轮转主题是 [[Evaluation]] / [[Judge Reliability]]。在当前 vault map 里，它是 RAG、Agent、Deep Research Agents 的共同测量层：如果 judge、reference、calibration 或 human trust protocol 本身不可靠，那么后续任何“系统更强”的结论都可能只是测量误差。

这轮不是 No Strong Signal。5 月上旬的新信号把 judge reliability 往前推进了一层：问题不再只是“LLM judge 有偏差”，而是 估计器、校准协议、解释呈现方式和人类信任行为都会制造新的评测误差。

本周位置

Adjacent topic。

本周锚点仍然是 RAG；Evaluation / Judge Reliability 不直接提升检索召回，但决定我们能否可信地判断 RAG 是否真的有证据、Agent 是否真的完成任务、Deep Research 报告是否真的可核验。

和 RAG 的关系

RAG 的承诺是“回答由外部证据支撑”。但这件事最后通常交给 judge、evaluator、citation checker 或人工 reviewer 判断。若 judge 校准不稳、reference 不可靠、解释让人产生 false trust，RAG 评测就可能把漂亮引用、顺滑推理或模型自信误读成真实证据质量。

本主题你现在需要知道的 3 件事

1. 校准不是一次性修补。

LLM-as-a-Judge 的 raw output 有偏，校准估计器也可能有偏；把一个模型上的校准结果共享给另一个模型，甚至可能让比较方向反转。

2. 解释可能更会说服人，而不是更能帮助判断。

reasoning trace、trace summary 和 post-hoc explanation 可能提高用户接受 AI 答案的概率，却不提高用户区分正确和错误的能力。

3. Judge reliability 正在变成端到端测量链问题。

prompt wording、style bias、reference quality、human grounding、calibration stability、用户信任行为都要进入评测协议，而不是只报告一个 judge agreement。

Key Patterns

1. 从 judge output 到 judge estimator。

Bias and Uncertainty in LLM-as-a-Judge Estimation 的关键不是再次证明 judge 有偏，而是说明 bias correction / calibration estimator 本身也可能失效。它提出用 judge quality J 和 cross-model calibration instability ΔJ 判断 corrected estimate 是否可靠。

2. 从 explanation 可信到 false trust 可测。

Evaluating the False Trust engendered by LLM Explanations 显示 reasoning traces 和 post-hoc explanations 更像 persuasion signal：它们会让用户更愿意接受答案，但不一定提高 correctness discrimination。

3. 从 prompt sensitivity 到 judge sensitivity。

JudgeSense 把语义等价 prompt 改写下 verdict 是否一致做成 benchmark。它提醒我们：如果评测 prompt 的等价改写都能改 verdict，评测结论就必须报告协议方差。

4. 从单点 debias 到 pipeline debias。

Judging the Judges 比较多种 debiasing strategy，最有用的发现是 style bias 可能远大于 position bias；这会直接影响 RAG / Agent 输出的自动评分。

5. 从 label correctness 到 construct validity。

本地 No Free Labels 说明没有 grounding 的 judge label 不能当真值；但即使有程序化 verifier，也还要问 benchmark 是否真的测到了目标能力，而不只是测格式服从或答案匹配。

Read Now

1. Bias and Uncertainty in LLM-as-a-Judge Estimation

  • why now： 这是 2026-05-07 的新提交，直接更新 judge reliability 的统计层问题。
  • what changed： 它把 naive judge output、bias-corrected estimator、shared calibration 的失效条件放在同一框架下，并展示模型比较中可能出现 sign reversal。
  • read it for： 判断什么时候 LLM-as-a-Judge 分数即使经过校准也不能相信。
  • how to start： 先读 J 和 ΔJ 的定义，再看 MMLU-Pro case study 中方向反转的例子。

2. Evaluating the False Trust engendered by LLM Explanations

  • why now： 它把 evaluation 从模型输出质量推进到用户如何被解释影响。对 human-in-the-loop judge 尤其重要。
  • what changed： 论文比较 reasoning traces、trace summaries、post-hoc explanations 和 dual explanation，发现单向解释常常提高接受率但不提高辨别能力，dual explanation 才真正改善判断。
  • read it for： 区分 explanation 的两个目标：让人更相信，还是让人更会判断。
  • how to start： 先看 false trust 的实验设置，再看 dual explanation 为什么比 reasoning trace 更有用。

3. JudgeSense: A Benchmark for Prompt Sensitivity in LLM-as-a-Judge Systems

  • why now： 它是 prompt sensitivity 和 judge reliability 的直接交叉，适合补上 05-06 已经出现的协议敏感性主线。
  • what changed： 用 494 组验证过的 paraphrase pairs 定义 Judge Sensitivity Score，显示部分 judge 在等价改写下 verdict 不稳，pairwise tasks 还可能出现 always-A 行为。
  • read it for： 给任何 LLM judge protocol 加一个最小稳定性检查。
  • how to start： 先读 JSS 定义，再看 factuality prompt artifact 和 pairwise always-A 结果。

本地底线论文。正确性任务中，没有可靠 reference、人类验证或 verifier 时，LLM judge output 不是免费真值。
